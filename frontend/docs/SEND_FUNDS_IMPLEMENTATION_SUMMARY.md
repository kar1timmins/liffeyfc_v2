## Key Clarifications

### Wallet Type Distinction
The application now clearly distinguishes between:

| Type | Generated By | Used For | User Access |
|------|-------------|----------|------------|
| **Liffey Account Wallet** | Backend at registration | Deploy bounties, receive funds | View-only, auto-managed |
| **MetaMask Wallet** | User's MetaMask extension | Sign transactions, send funds | User controls fully |

### Why Both Matter
- **Liffey Wallet**: Enables automated bounty deployments without user action
- **MetaMask Wallet**: Enables user to prove ownership and send their own funds
- **Distinction**: Prevents confusion and clarifies what each is used for

## API Integration

### Wallet Balance Endpoint
```
GET /wallet-balance?address={address}&chain={ethereum|avalanche}
Returns: { balanceWei, balanceEth/balanceAvax, rpcEndpoint }
```

### Gas Price Endpoint
```
GET /wallet-balance/gas-price?chain={chain}
Returns: { gasPriceGwei, rpcEndpoint }
```

**Backend Implementation**:
- Multiple RPC fallbacks for reliability
- CORS proxy prevents frontend RPC calls
- Used by SendFunds for real-time data

## Security Considerations

✅ **Wallet Connection**: MetaMask-based (user controls keys)
✅ **Transaction Signing**: User signs all transactions
✅ **Balance Display**: Read-only, no fund access
✅ **Address Validation**: Checks 0x + 40 hex format
✅ **Network Safety**: Users must switch to correct network first
